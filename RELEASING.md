# Releasing Ralph

> This document is the single source of truth for how Ralph's three artifacts are
> released. It supersedes `docs/PUBLISHING.md`.

Ralph ships three independently versioned components:

| Component       | Path                      | Artifact                       | Tag schema             |
| --------------- | ------------------------- | ------------------------------ | ---------------------- |
| `ralph-core`    | `packages/core`           | npm `@daonhan/ralph-core`      | `ralph-core-vX.Y.Z`    |
| `ralph`         | `apps/cli`                | npm `@daonhan/ralph`           | `ralph-vX.Y.Z`         |
| `ralph-sandbox` | `packages/core/templates` | Docker `daonhan/ralph-sandbox` | `ralph-sandbox-vX.Y.Z` |

Releases are driven by [release-please](https://github.com/googleapis/release-please)
in manifest mode. The config lives at the repo root in `release-please-config.json`
and `.release-please-manifest.json` (the manifest is the source of truth for the
last-released version of each component). Conventional-commit history on `main`
determines what gets released; you write `feat:` / `fix:` commits as usual and a
Release PR appears.

## 2. How to cut a release

1. Land your work on `main` with Conventional Commits (see §3). No version bump by
   hand — release-please computes it.
2. release-please opens (or updates) **one Release PR per component** that has
   unreleased commits. The PR bumps the version, amends `CHANGELOG.md`, and carries
   the refreshed status table.
3. Review the PR: confirm the proposed version and the CHANGELOG diff read the way
   you want users to see them.
4. **Merge the Release PR.** That single action creates the component tag
   (`<component>-vX.Y.Z`) and the GitHub Release.
5. The tag triggers the matching publish workflow:
   - `ralph-core-v*` / `ralph-v*` → `publish-npm.yml` → publishes to npm, then
     attaches the `.tgz`, SBOM, and cosign attestation to the Release.
   - `ralph-sandbox-v*` → `publish-image.yml` → builds + pushes the image, pins the
     `sha256:` digest into the Release body, and attaches the image SBOM +
     attestation.
6. Watch the Actions tab; the publish workflow run links to the published artifact.

> **Token requirement.** release-please must create tags with a Personal Access
> Token (`RELEASE_PLEASE_TOKEN` secret, repo scope), because a tag created with the
> default `GITHUB_TOKEN` does **not** trigger the `push: tags` publish workflows.
> Required secrets: `RELEASE_PLEASE_TOKEN`, `NPM_TOKEN`, `DOCKERHUB_USERNAME`,
> `DOCKERHUB_TOKEN`. If only `GITHUB_TOKEN` is available, merge still creates the
> tag but you must start the publish manually (`workflow_dispatch` on the relevant
> workflow with the tag name).

## 3. Conventional Commit guide

The commit type drives both the version bump and the CHANGELOG section. Convention
is unchanged from the existing history — adopting release-please required no
contributor behavior change.

| Prefix      | Bump (0.x)      | CHANGELOG section         | Triggers a release? |
| ----------- | --------------- | ------------------------- | ------------------- |
| `feat:`     | minor           | Features                  | yes                 |
| `fix:`      | patch           | Bug Fixes                 | yes                 |
| `perf:`     | patch           | Performance Improvements  | yes                 |
| `deps:`     | patch           | Dependencies              | yes                 |
| `refactor:` | patch           | Code Refactoring          | yes                 |
| `chore:`    | —               | Miscellaneous Chores      | no (changelog only) |
| `ci:`       | —               | (hidden)                  | no                  |
| `docs:`     | —               | (hidden)                  | no                  |
| `test:`     | —               | (hidden)                  | no                  |
| `style:`    | —               | (hidden)                  | no                  |

Breaking changes: append `!` (e.g. `feat!:`) or add a `BREAKING CHANGE:` footer. See
§4 for how breaking changes map to versions under this repo's policy.

Which component a commit belongs to is decided by the path it touches:

- `packages/core/**` (excluding `templates/`) → `ralph-core`
- `apps/cli/**` → `ralph`
- `packages/core/templates/**` (includes the `Dockerfile`) → `ralph-sandbox`

## 4. Version policy

- **Independent versions per component.** The three components are never released in
  lockstep; a change to one does not force a bump of the others.
- **0.x is treated as a stable contract.** Stricter than the SemVer spec's "0.x means
  anything goes": while on 0.x, a backward-compatible feature is a **minor** bump, a
  bug fix is a **patch** bump, and any **breaking change forces `1.0.0`**. Downstream
  users can pin a 0.x version and trust that a patch or minor upgrade will not break
  them.
- **Enforcing the breaking→1.0.0 rule.** release-please will not auto-jump from 0.x to
  1.0.0 on a breaking change (pre-1.0 it bumps the minor). When a breaking change
  lands, the maintainer forces the major with a `Release-As:` footer (§5):
  `Release-As: 1.0.0`.

## 5. Pre-release tags and version overrides

Pre-release tags (`-rc.N`, `-next.N`) are permitted but off by default. To opt in for
a single release, add a `Release-As:` footer to a commit on `main`; release-please
honors it for the next Release PR of the component that commit touches.

```
feat: gate the new reviewer stage behind a flag

Release-As: 0.3.0-rc.1
```

The same footer is how you force any exceptional version — e.g. the breaking→1.0.0
jump from §4:

```
feat!: drop the legacy RALPH_IMAGE_TAG alias

BREAKING CHANGE: RALPH_IMAGE_TAG is removed; use RALPH_IMAGE.
Release-As: 1.0.0
```

## 6. Rollback runbook

Act fast; don't invent a process during an incident.

### npm (`@daonhan/ralph-core`, `@daonhan/ralph`)

npm does not allow unpublishing after 72h (or once another package depends on the
version). Deprecate the bad version(s) so installs surface a warning, then ship a
fixed version through the normal flow.

```bash
# Deprecate each affected version with a message pointing at the fix.
npm deprecate @daonhan/ralph-core@0.1.2 "Broken release — upgrade to 0.1.3"
# A range works too:
npm deprecate "@daonhan/ralph@>=0.2.0 <0.2.2" "Broken release — upgrade to 0.2.2"

# Then cut a fixed release the normal way (commit a fix:, merge the Release PR).
# To clear a deprecation after the fix lands:
npm deprecate @daonhan/ralph-core@0.1.2 ""
```

### Docker (`daonhan/ralph-sandbox`)

A digest is immutable, so "rollback" means repointing the moving tags (`:latest`,
`:vX.Y.Z`) away from the bad build, and/or deleting the bad tag.

```bash
# Repoint :latest (and the version tag) at the last-known-good digest without a
# rebuild. Use the digest recorded in the previous good GitHub Release body.
docker buildx imagetools create \
  -t docker.io/daonhan/ralph-sandbox:latest \
  docker.io/daonhan/ralph-sandbox@sha256:<good-digest>

# Optionally delete the bad tag via the Docker Hub API (token = DOCKERHUB_TOKEN):
curl -s -X DELETE \
  -H "Authorization: JWT <token>" \
  "https://hub.docker.com/v2/repositories/daonhan/ralph-sandbox/tags/v0.1.2/"
```

For an emergency rebuild that does not warrant a semver bump (e.g. a base-image CVE),
run `publish-image.yml` via `workflow_dispatch` with the tag to rebuild.

### GitHub Release

Edit the affected Release to mark it clearly (prefix the title with `[YANKED]`) and
note the replacement version in the body, so the paper trail stays honest.

## 7. Compatibility matrix

Known-good combinations. Update this when a release changes a compatibility
boundary (e.g. the CLI starts requiring a newer core).

| `@daonhan/ralph` (CLI) | requires `@daonhan/ralph-core` | tested `ralph-sandbox` image |
| ---------------------- | ------------------------------ | ---------------------------- |
| `0.1.0`                | `^0.1.1`                       | `v0.1.1` / `latest`          |

The CLI declares its core dependency as `workspace:^` in the repo; the publish
pipeline rewrites that to the concrete `^<core-version>` it was built against, so an
installed `@daonhan/ralph` always pulls a compatible `@daonhan/ralph-core`.

## 8. Tag naming and the synthetic `ralph-sandbox` component

Each component owns a tag prefix so each publish workflow matches only its own
releases:

| Component       | Tag                    | Workflow filter        |
| --------------- | ---------------------- | ---------------------- |
| `ralph-core`    | `ralph-core-vX.Y.Z`    | `ralph-core-v*`        |
| `ralph`         | `ralph-vX.Y.Z`         | `ralph-v*`             |
| `ralph-sandbox` | `ralph-sandbox-vX.Y.Z` | `ralph-sandbox-v*`     |

`ralph-sandbox` is **synthetic**: it has no `package.json` (release-type `simple`).
It is rooted at `packages/core/templates`, which is where the image's inputs live —
the `Dockerfile` and the agent playbook/iteration templates. `ralph-core` is rooted
at `packages/core` and **excludes** `packages/core/templates`, so the two never bump
each other:

- a change under `packages/core/src/**` bumps `ralph-core` only;
- a change to `packages/core/templates/**` (including the `Dockerfile`) bumps
  `ralph-sandbox` only.

> **Why the `Dockerfile` lives under `templates/`.** release-please attributes each
> changed file to a component by longest-matching **directory** prefix and has no
> per-file/glob include scoping. Two components cannot share the `packages/core`
> root, and a file sitting next to `src/` cannot be routed to a different component.
> Moving the `Dockerfile` into `templates/` (the directory the synthetic component
> owns) is what makes the Dockerfile-only bump possible. The runtime build-fallback
> resolves the Dockerfile from `templates/Dockerfile` first, falling back to the
> context root for older layouts.

The existing `image-v0.1.1` tag and the already-published 0.x npm versions are
preserved; nothing is re-published or moved. `publish-image.yml` keeps the legacy
`image-v*` trigger for one release as a compatibility shim, removed in a follow-up.
